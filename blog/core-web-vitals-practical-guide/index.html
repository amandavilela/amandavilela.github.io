A practical guide to Core Web Vitals

April 10, 2025

Core Web Vitals are Google's set of real-user experience metrics. They measure three specific aspects of how a page feels in practice: loading, visual stability, and responsiveness. Getting them right matters both for users and for search rankings.

The three metrics

LCP — Largest Contentful Paint

LCP measures how long it takes for the largest visible element on the page to render. That element is usually a hero image, a large heading, or a banner.

A good LCP is under 2.5 seconds. Above 4 seconds is considered poor.

The most common causes of a slow LCP are:

  • Large, unoptimised images
  • Render-blocking scripts or stylesheets in <head>
  • Slow server response times (high TTFB)
  • Late-discovered resources (images loaded via CSS or JavaScript)

Start by identifying your LCP element in Chrome DevTools (Performance panel → LCP marker). Then check whether the resource is preloaded, whether it is served in a modern format like WebP or AVIF, and whether it has explicit width and height attributes to avoid layout shifts while it loads.

CLS — Cumulative Layout Shift

CLS measures visual stability — how much content unexpectedly jumps around while the page loads.

A good CLS score is under 0.1. Anything above 0.25 is poor.

The most common culprits are:

  • Images and iframes without explicit dimensions
  • Late-injected banners or cookie notices
  • Web fonts that cause text to reflow when they load (FOIT/FOUT)

The fix is almost always the same: reserve space. Set width and height on every image. Use font-display: optional or size-adjust on web fonts. If you must inject dynamic content, insert it below the fold.

INP — Interaction to Next Paint

INP replaced FID in March 2024. It measures the worst interaction delay across the full page visit — not just the first one.

A good INP is under 200 milliseconds. Above 500 milliseconds is poor.

Heavy JavaScript is the main driver of poor INP. Long tasks block the main thread and delay the browser from painting the response to a user interaction. Break up long tasks with scheduler.yield() or setTimeout, defer non-critical scripts, and audit third-party tags — they are often the biggest offenders.

A simple diagnostic workflow

  1. Run a field data check in PageSpeed Insights to see real-user data from the Chrome User Experience Report.
  2. Open the Performance panel in Chrome DevTools and record a page load. Look for the LCP marker and any long tasks (red blocks).
  3. Use the Interactions track in the Performance panel to find slow INP candidates.
  4. Check the Layout Shifts track for unexpected shifts and identify which elements are moving.

Where to start

If you only have time to fix one thing, fix LCP. It has the highest correlation with user perception of speed and tends to have the clearest path to improvement. Compress and lazy-load images below the fold, preload the hero image, and reduce your TTFB with caching or a CDN.

CLS and INP improvements often follow naturally once you have a disciplined approach to loading assets and managing JavaScript execution.
