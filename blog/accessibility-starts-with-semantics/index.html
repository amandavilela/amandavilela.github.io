Accessibility starts with semantics

May 20, 2025

A common misconception in frontend development is that accessibility is mostly about ARIA. Developers add role="button" to a <div>, sprinkle in some aria-label attributes, and consider the job done. But ARIA is a patch, not a foundation. The foundation is semantic HTML.

What semantics actually do

When you use the right HTML element, the browser does a lot of work for you. A <button> is focusable by default, activatable with both Enter and Space, and announced as a button by screen readers. A <div> with role="button" needs all of that behaviour wired up manually — and it's easy to miss something.

The same applies across the board:

  • <nav> creates a landmark region screen reader users can jump to directly
  • <h1>–<h6> give users a document outline they can navigate by heading
  • <label> associates text with a form control without any JavaScript
  • <table> communicates row and column relationships that a styled <div> grid never can

The ARIA first rule

The first rule of ARIA use is: don't use ARIA if a native HTML element or attribute already has the semantics you need. This isn't an opinion — it's in the spec.

ARIA is designed to fill gaps where native HTML falls short, not to replace it. Using role="list" on a <ul> that already has list semantics adds noise. Using aria-required="true" on an <input required> is redundant. These things are harmless individually, but they signal a workflow where semantics are an afterthought.

An example: interactive cards

Consider a grid of clickable cards. A common pattern looks like this:

<div class="card" onclick="navigate()">
  <div class="card__title">Audit</div>
  <div class="card__desc">Clear diagnosis and prioritized action plan.</div>
</div>

This fails in several ways. It is not keyboard accessible. Screen readers announce it as nothing — just a group of text. You would need role="button", tabindex="0", onkeydown for Enter and Space, and an accessible name. That is a lot of manual work to recreate what you get for free with:

<a href="/services/audit" class="card">
  <h3 class="card__title">Audit</h3>
  <p class="card__desc">Clear diagnosis and prioritized action plan.</p>
</a>

The <a> is keyboard focusable, activatable with Enter, and announced as a link with its heading text as the label. The <h3> gives it structure in the document outline. No ARIA needed.

Semantics and automated testing

Another benefit of semantic HTML is that automated accessibility tools — axe, Lighthouse, WAVE — are much better at catching issues in semantic markup. When you use custom elements and ARIA patches, tooling has less to work with and misses more.

That said, automated tools catch roughly 30–40% of accessibility issues. The rest require manual testing and screen reader use. Semantic HTML helps with the manual layer too: a well-structured document is far easier to navigate with a keyboard or screen reader than a flat collection of <div> elements.

Where to start

If you are auditing an existing codebase, a useful first pass is to look for:

  • Interactive <div> or <span> elements that should be <button> or <a>
  • Form fields without associated <label> elements
  • Images without meaningful alt text
  • Heading levels that skip or are used for styling rather than structure
  • Lists rendered as <div> elements

These are quick wins. They do not require a design change, just the right element. And they improve the experience not just for screen reader users, but for keyboard users, voice control users, and anyone on a slow or unfamiliar device.

Semantics are not a feature to add at the end. They are the material the page is built from.
